## (2) Security-relevant decisions

- **CI Claude-Review removal is a policy change, not a vulnerability introduction.** Review moves from CI-blocking to local honor-system. Reactivation triggers documented in `tests.yml` header + `CLAUDE.md`. Rollback is a single-commit revert + `scripts/apply-branch-protection.sh` to re-add the three required status checks.
- **Branch-protection snapshot → apply-script pattern.** Prior attempt at `gh api --input` with a decorated JSON file would 4xx (wrapper keys `_comment`, `notes`, etc. aren't part of the API contract). The new `scripts/apply-branch-protection.sh` uses `jq '.rules'` to extract the API-shaped payload, prompts for confirmation, and verifies the applied state. No untrusted-input flow; `set -euo pipefail`; quoted variable expansion throughout.
- **Chirpy fork surface grew** by two files (`topbar.html`, `head.html`). Both carry provenance comments pinning the upstream version, so a `bundle update` that bumps Chirpy produces a discoverable re-diff target.
- **Mobile search-focus hide rule** uses CSS `:has(#search.input-focus)` — Chirpy's own toggle class — to hide the switcher when search expands on mobile. No new attack surface; CSS compensation only.
- **No new dependencies** in `Gemfile` or `package.json`.
- **No new workflow permissions.** Permissions block unchanged; removing jobs subtracts surface rather than adding it.

### Why zero

- **No untrusted input surfaces added.** Static site; no form handlers, no query-param interpolation, no new API endpoints.
- **No dependency bumps.** Gemfile / package.json unchanged; lockfiles untouched.
- **No new write permissions** in any workflow job. Removing jobs shrinks the attack surface.
- **Fork files are Liquid-only.** All interpolated values repo-controlled (frontmatter, `_data/i18n.yml`, Chirpy locale files). Liquid auto-escapes; no `| raw` or `| safe` used.
- **`apply-branch-protection.sh`** uses strict mode, `read -r -p` confirmation, quoted variable expansion, and a `jq` → `gh api` pipeline with no shell interpolation of untrusted data. Trusted inputs (`$1` optional repo name) per the trust-model rules.
- **Docs + tests** excluded per the skill's rules 11 and 16.

The CI policy weakening (removing Claude Review gating) is a defense-in-depth reduction, not a vulnerability. It's documented with explicit reactivation triggers and a preserved rollback path.
